## 1. 프로젝트 개요

본 프로젝트는 RPG 전투 시뮬레이션을 반복 실행하여 목표 승률을 만족하도록 몬스터 스탯을 자동 역산하는 데이터 기반 밸런스 최적화 시스템이다. RPG 게임은 이 엔진을 검증하기 위한 실험 환경이며, 핵심은 시뮬레이션과 로그 데이터를 기반으로 밸런싱 의사결정을 자동화하는 구조이다.

기존 게임 밸런싱은 개발자가 직접 플레이 테스트를 반복하며 수작업으로 수치를 조정해야 한다. 이 방식은 시간이 오래 걸리고, 경우의 수가 많아 최적값을 찾기 어려우며, 플레이어 상태를 반영하기 어렵다. 본 시스템은 Monte Carlo 시뮬레이션과 Binary Search를 조합하여 이 문제를 자동화한다.

| 구분 | 기존 방식 | 본 시스템 |
|---|---|---|
| 밸런스 조정 | 개발자 수동 플레이 테스트 | 시뮬레이션 기반 자동 역산 |
| 승률 편차 | 조합에 따라 크게 변동 | 목표 승률에 수렴하도록 반복 조정 |
| 조정 소요 시간 | 수십 분 ~ 수 시간 | 수 초 ~ 수십 초 수준 |
| 플레이어 상태 반영 | 불가능 또는 제한적 | PlayerPowerIndex 기반 보정 |

## 2. 핵심 설계 원칙

본 시스템은 세 가지 설계 원칙을 중심으로 구성된다.

### 2.1 Digital Twin

시뮬레이션과 실제 전투가 동일한 `Battle_Engine`을 공유한다. 시뮬레이션 전용 엔진을 별도 구현하면 실제 전투와 시뮬레이션 결과 간 오차가 발생한다. 동일 엔진을 사용함으로써 밸런싱 결과가 실제 게임에서도 최대한 동일한 규칙으로 적용된다.

### 2.2 Decoupling

UI / Engine / AI 레이어를 분리하였다. `Action_Class.py`는 전투 UI를 담당하고, `Battle_Engine.py`는 순수 전투 로직을 처리하며, `Auto_AI.py`는 규칙 기반 전투 판단을 담당한다. 이 구조 덕분에 Flask 기반 웹 경로로 확장하더라도 핵심 전투 엔진을 재사용할 수 있다.

### 2.3 Non-blocking

밸런싱 시뮬레이션을 백그라운드 스레드로 분리하여 게임 루프가 중단되지 않는다. 게임 시작 시 고블린, 박쥐 2종 시뮬레이션을 백그라운드로 실행하고, 완료 전 전투 요청이 들어오면 폴백 스탯으로 즉시 응답한 뒤 완료 시점에 캐시를 교체한다.

## 3. 전체 시스템 흐름

플레이어가 전투를 시작하면 `BalanceHook`이 백그라운드에서 몬스터 밸런싱을 자동 실행한다.

| 모듈 | 역할 | 핵심 설계 |
|---|---|---|
| `Battle_Engine.py` | 전투 판정 엔진 | ATB 시스템 / 데미지 공식 / 버프·디버프 |

## 4. 직업별 스킬 시스템과 경우의 수

밸런싱 문제의 복잡도는 플레이어가 보유한 스킬 조합에 따라 달라진다. 본 시스템은 직업별 고정 스킬 트리를 설계하여 시뮬레이션 케이스를 구조화한다.

### 4.1 직업별 스킬 트리

4개 직업(전사 / 마법사 / 탱커 / 도적)은 레벨업 시 자동으로 해당 직업 스킬을 습득한다. 상위 스킬을 습득하면 하위 스킬이 자동 대체된다.

| 직업 | 스킬 개수 | 주요 스킬 유형 | 레벨 범위 |
|---|---:|---|---|
| 전사 | 10개 | 물리 공격, 버프, 디버프 | Lv1~20 |
| 마법사 | 12개 | 마법 공격, 힐, 디버프 | Lv1~25 |
| 탱커 | 9개 | 방어 공격, 실드, 수비태세, 저주 | Lv1~18 |
| 도적 | 8개 | 급소찌르기, 연속찌르기, 난사, 둔화 | Lv1~14 |

### 4.2 스킬 경우의 수가 밸런싱에 미치는 영향

플레이어가 보유한 스킬 조합은 `PlayerPowerIndex` 계산에 직접 반영된다. `Simulator.py`의 `PlayerPowerIndex._skill_expected_dmg()` 메서드는 보유 스킬 중 최고 기대 기여값을 계산한다.

```python
physical   : player.stg * mult * hits
magical    : player.sp  * mult * hits
multi_hit  : player.stg * expected_hits * dmg_decay
tank_attack: player.arm * arm_mult + player.maxhp * hp_mult
heal       : (base_heal + sp * sp_mult) * 0.6
buff       : base_stat * buff_amount * turns * 0.5
shield     : maxhp * shield_mult * 0.5
```

이 값이 클수록 `PlayerPowerIndex`가 높아지며, 목표 승률이 낮아져 더 강한 몬스터가 생성된다. 즉 스킬을 많이 보유하고 MP가 충분할수록 게임이 자동으로 더 어렵게 조정된다.

### 4.3 스킬 메타데이터 구조

모든 스킬은 `SKILL_META` 딕셔너리에 정의된다. 시뮬레이션 AI와 실제 전투 AI가 동일 메타데이터를 참조하여 Digital Twin 원칙을 유지한다.

| 스킬 타입 | 예시 스킬 | 핵심 파라미터 |
|---|---|---|
| `physical` | 강타1/2, 슬래시1/2 | `mult`, `hits`, `mp` |
| `magical` | 파이어볼1/2, 아이스볼릿, 라이트닝 | `mult`, `hits`, `mp` |
| `multi_hit` | 연속공격1/2, 연속찌르기, 난사 | `base_prob`, `luc_mult`, `max_hits`, `dmg_decay` |
| `debuff` | 약화, 마약화, 저주, 둔화 | `debuff_stat`, `debuff_amount`, `debuff_turns` |
| `buff` | 강화, 효율성, 수비태세, 추진력 | `buff_stat`, `buff_amount`, `buff_turns` |
| `heal` | 힐1/2 | `base_heal`, `sp_mult` |
| `shield` | 실드 | `shield_mult` |
| `counter` | 되갚기1/2 | `counter_mult`, `arm_mult`, `cap` |
| `tank_attack` | 몸통박치기1/2 | `arm_mult`, `hp_mult` |

## 5. 자동 밸런싱 알고리즘

자동 밸런싱은 Monte Carlo 시뮬레이션으로 승률을 측정하고, Binary Search로 목표 승률에 도달하는 몬스터 스탯 배율(`scale`)을 역산하는 방식으로 구성된다.

### 5.1 전체 알고리즘 흐름

| 단계 | 내용 | 현재 구현 기준 |
|---|---|---|
| 1 | 목표 승률 설정 | 몬스터 종류별 기본 목표 승률 + PlayerPowerIndex 보정 |
| 2 | 시뮬레이션 실행 | 현재 scale로 N회 전투 반복 |
| 3 | 승률 비교 | 목표 승률과 현재 승률 오차 확인 |
| 4 | scale 조정 | 이진 탐색으로 범위 축소 |
| 5 | 최종 검증 | 수렴된 scale로 추가 반복 검증 |

현재 구현 기준:
- `SIM_N = 300`
- 허용 오차 `±0.03`
- 최대 반복 `20회`
- 최종 검증 `500회`

### 5.2 Binary Search 수렴 과정

이진 탐색은 `lo=0.6`, `hi=5.0` 범위에서 시작한다.  
각 반복에서 `mid = (lo + hi) / 2`를 계산하고, 해당 scale로 생성한 몬스터를 300회 시뮬레이션하여 승률을 측정한다.

```python
lo, hi = 0.6, 5.0
for _ in range(MAX_ITER):
    mid   = (lo + hi) / 2
    enemy = _scale_enemy(mid)
    sim   = BattleSimulator(player, enemy, n=300).run()

    if abs(sim.win_rate - target) <= 0.03:
        break

    if sim.win_rate > target:
        lo = mid
    else:
        hi = mid
```

현재 구현은 목표 승률 ±3% 이내 수렴을 목표로 설계되어 있다. 다만 실제 최신 전투 수치 체계에서는 일부 구간, 특히 초반 구간에서 추가 튜닝이 필요할 수 있다.

### 5.3 스탯 스케일 공식

모든 스탯에 동일한 비율을 곱하면 HP가 지나치게 커져 전투가 과도하게 길어질 수 있다. 이를 방지하기 위해 스탯별로 다른 공식을 적용한다.

```python
e.hp  = e.hp  * (0.60 + 0.30 * scale)
e.stg = e.stg * sqrt(scale)
e.arm = e.arm * min(scale, 1.8)
e.luc = e.luc * min(scale, 1.8)
```

| 스탯 | 공식 | 특징 |
|---|---|---|
| HP | `base * (0.60 + 0.30*scale)` | 선형 스케일보다 완만하게 증가 |
| STG | `base * sqrt(scale)` | 폭딜 방지 |
| ARM | `base * min(scale, 1.8)` | 상한 제한 |
| LUC | `base * min(scale, 1.8)` | 상한 제한 |

예시:
- `scale=0.6` → HP `×0.78`
- `scale=1.0` → HP `×0.90`
- `scale=3.0` → HP `×1.50`
- `scale=5.0` → HP `×2.10`

## 6. PlayerPowerIndex — 동적 난이도 조정

플레이어의 현재 상태를 수치화하여 목표 승률을 실시간으로 보정한다. 같은 `normal` 난이도라도 플레이어가 HP가 낮고 포션이 없으면 목표 승률이 올라가고, 반대로 HP가 높고 스킬이 강하면 목표 승률이 내려간다.

### 6.1 지수 계산 구조

지수는 `1.0`을 기준선으로 시작해 다음 요소를 더하거나 빼는 방식으로 계산된다. 최종값은 `0.4 ~ 2.0` 범위로 제한된다.

| 반영 항목 | 조건 | 지수 변화 | 의미 |
|---|---|---:|---|
| HP 비율 | HP ≥ 80% | +0.15 | 체력 충분 |
| HP 비율 | HP 50~79% | +0.00 | 보통 |
| HP 비율 | HP 30~49% | -0.15 | 체력 위험 |
| HP 비율 | HP < 30% | -0.30 | 위급 |
| MP + 스킬 | MP ≥ 70% + 스킬 보유 | +0.20 | 스킬 활용 여유 |
| MP + 스킬 | MP 40~69% | +0.10 | 보통 |
| MP + 스킬 | MP < 20% | -0.10 | 스킬 활용 저하 |
| 아이템 | HP 포션 3개 이상 | +0.15 | 회복 여유 |
| 아이템 | HP 포션 1개 이상 | +0.07 | 보통 |
| 아이템 | HP 포션 없음 | -0.10 | 불리 |
| 스탯 수준 | STG 또는 SP ≥ 15 | +0.10 | 고스탯 |
| 스탯 수준 | STG와 SP 모두 ≤ 5 | -0.10 | 저스탯 |
| SPD | SPD ≥ 14 | +0.15 | 고속 직업 |
| SPD | SPD ≥ 12 | +0.08 | 비교적 빠름 |
| SPD | SPD ≤ 7 | -0.08 | 저속 |

### 6.2 목표 승률 보정 공식

```python
delta  = (1.0 - power_index) * 0.08
target = base_target + delta
target = max(0.20, min(0.97, target))
```

즉 현재 구현에서 목표 승률 보정 폭은 약 `±8%p` 수준이다.

예시:
- `PowerIndex = 1.35`
- `delta = (1.0 - 1.35) * 0.08 = -0.028`
- 고블린 normal 기본 목표 승률 `0.60` → 보정 목표 승률 약 `0.572`

또한 몬스터 종류에 따라 기본 목표 승률을 다르게 둔다.

| 몬스터 | hard | normal | easy |
|---|---:|---:|---:|
| 고블린 | 0.45 | 0.60 | 0.70 |
| 박쥐 | 0.80 | 0.90 | 0.95 |

박쥐는 유리대포형 몬스터로 설계되어 플레이어 승률이 높도록 별도 목표를 사용한다.

## 7. 데미지 계산 공식 (DamageCalc)

`Battle_Engine.py`의 `DamageCalc` 클래스는 물리/마법 데미지를 단일 공통 함수 `_calc()`로 처리한다. 시뮬레이션과 실제 전투 모두 동일 함수를 사용한다.

### 7.1 계산 순서

```text
Damage = Base × skill_mult × role_mult × random_factor × crit
```

| 단계 | 공식 | 비고 |
|---|---|---|
| 회피 판정 | `evade_chance = min(def_luc × 0.4, 25)` | 최대 25% |
| Base 데미지 | `atk_stat × 250 / (100 + def_stat)` | 방어력 비율 적용 |
| 최소 보장 | `max(base, atk_stat × 0.4)` | 0 데미지 방지 |
| 스킬 계수 | `base × skill_mult` | `SKILL_META` 참조 |
| 역할 배율 | `base × role_mult` | 현재 플레이어/몬스터 모두 1.0 |
| 랜덤 요소 | `base × uniform(0.9, 1.1)` | ±10% |
| 크리티컬 | `crit_chance = min(atk_luc × 0.5, 40)` 후 ×1.5 | 최대 40% |

### 7.2 시뮬레이션-실전 일관성

`Auto_AI`가 시뮬레이션에서 `DamageCalc`를 호출하는 방식과 `Action_Class`가 실제 전투에서 호출하는 방식은 동일 엔진을 공유한다. 이것이 Digital Twin 원칙의 핵심 구현이다.

## 8. 규칙 기반 전투 AI (Auto_AI.py)

시뮬레이션의 신뢰도는 AI가 얼마나 현실적인 우선순위로 행동하느냐에 달려 있다. `Auto_AI.py`는 `PlayerAI`와 `EnemyAI`로 구성되며, 우선순위 기반 규칙으로 행동을 결정한다.

### 8.1 PlayerAI 우선순위 예시

| 우선순위 | 조건 | 행동 |
|---|---|---|
| 1 | HP 35% 이하 + 힐 가능 | 힐 스킬 |
| 2 | HP 위험 구간 | HP 포션 |
| 3 | 속도 열세 | 둔화 디버프 |
| 4 | 필요 시 | 버프 / 디버프 |
| 5 | MP 여유 있음 | 효율 높은 공격 스킬 |
| 6 | MP 부족 | MP 포션 또는 기본 공격 |

### 8.2 스킬 효율 점수

`Auto_AI.py`는 MP 비용 대비 기대 성능을 기준으로 스킬 효율을 계산한다.

예:

```python
efficiency = (stat * mult * hits) / mp_cost
```

SPD 디버프 같은 일부 스킬은 전략적 가치를 반영해 별도 보정이 들어간다.

| 지원 직업 | 4종 | 전사 / 마법사 / 탱커 / 도적 |
| 총 스킬 수 | 39개 | 직업별 8~12개 |

주의할 점은, 현재 구현은 목표 승률에 수렴하도록 설계되어 있으나 실제 최신 전투 수치 체계에서는 일부 구간에서 추가 튜닝이 필요할 수 있다는 것이다.

## 10. 한계와 향후 계획

### 10.1 현재 한계

- 현재 시스템은 직업, 스킬, HP/MP, 아이템, SPD 등 주요 변수는 반영하지만, 실제 인간 플레이어의 비정형 행동까지 완전하게 모델링하지는 않는다.
- 규칙 기반 AI는 설명 가능성이 높지만, 모든 플레이 스타일을 완전하게 대체하지는 못한다.
- 일부 초반 구간에서는 목표 승률 수렴력이 약해 추가 밸런싱 조정이 필요할 수 있다.

### 10.2 향후 계획

| 시기 | 목표 | 세부 내용 |
|---|---|---|
| 1학기 (~6월 중순) | 콘텐츠 확장 | 몬스터 종류 확장, 보스 패턴 추가 |
| 방학 (6~8월) | 웹 서비스 배포 | Flask API 서버화, 웹 UI, 배포 환경 구성 |
| 2학기 (9~11월) | 데이터 기반 고도화 | 실사용자 로그 수집, 플레이 패턴 분석, LLM 보조 피드백 |

## 결론

본 프로젝트는 RPG 전투를 실험 환경으로 사용하여, Monte Carlo 시뮬레이션과 Binary Search를 기반으로 몬스터 밸런스를 자동 조정하는 시스템을 구현하였다. 또한 PlayerPowerIndex를 통해 플레이어 상태를 반영한 동적 난이도 조정을 시도하였고, 로그 저장 및 행동 분석 구조를 통해 향후 데이터 기반 고도화가 가능한 기반을 마련하였다.

핵심 의미는 단순한 게임 제작이 아니라, 시뮬레이션과 데이터를 이용해 밸런싱 의사결정을 자동화하는 엔진을 설계했다는 데 있다.
